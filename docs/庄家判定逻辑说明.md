 # 庄家判定逻辑说明

## 一、_run_dealer_check 方法概述

`_run_dealer_check` 是庄家判定的核心方法，负责判断一个钱包地址是否为"庄家"。

## 二、方法流程图

```
接收参数: address, mint, sig
     │
     ▼
┌─────────────────────────────────────────────┐
│ 1. 获取 Helius API Key                       │
│    - 从 SQLite settings 表读取              │
│    - 未配置 → 跳过判定，返回                 │
└─────────────────────────────────────────────┘
     │
     ▼
┌─────────────────────────────────────────────┐
│ 2. 获取钱包最旧一笔交易                       │
│    - 调用 _fetch_first_transaction           │
│    - 无交易记录 → 跳过，返回                 │
└─────────────────────────────────────────────┘
     │
     ▼
┌─────────────────────────────────────────────┐
│ 3. 读取已满足的条件列表                      │
│    - 从 Redis 读取该地址的 conditions         │
│    - 格式: ["C001", "C002"]                 │
└─────────────────────────────────────────────┘
     │
     ▼
┌─────────────────────────────────────────────┐
│ 4. 执行条件检测                              │
│                                             │
│    条件 C001:                                │
│    - 首笔交易包含 closeAccount 指令          │
│    - 且交易后该地址 SOL 余额为 0             │
│    - 满足 → 添加到 conditions 列表           │
│                                             │
│    后续条件可追加: C002, C003...             │
└─────────────────────────────────────────────┘
     │
     ▼
┌─────────────────────────────────────────────┐
│ 5. 判定结果                                  │
│                                             │
│    有满足的条件?                              │
│    ├─ 是 → 设置状态为 "dealer"              │
│    │         广播 WS 消息到前端               │
│    │         更新 Redis                      │
│    │                                          │
│    └─ 否 → 设置状态为 "unknown"              │
│              更新 Redis                      │
└─────────────────────────────────────────────┘
```

## 三、参数说明

| 参数 | 类型 | 说明 |
|------|------|------|
| address | str | 待检测的钱包地址 |
| mint | str | 当前监控的代币地址（用于 WS 广播） |
| sig | str | 当前触发检测的交易签名（用于前端匹配） |

## 四、Redis 数据结构

```
键: user:{address}
值: Hash
  - status: "retail" | "dealer" | "unknown"
  - conditions: JSON 列表，如 '["C001"]'
```

## 五、庄家条件说明

### 条件 C001：关闭账户模式

**判定规则：**
1. 钱包的第一笔交易包含 `closeAccount` 指令
2. 交易后该钱包的 SOL 余额为 0

**业务含义：**
- 庄家在发币前通常会创建临时账户
- 发币后会关闭这些临时账户，取回 SOL
- 关闭账户后余额归零是典型特征

## 六、WebSocket 广播

当判定为庄家时，会广播以下消息：

```json
{
    "type": "user_status",
    "data": {
        "address": "钱包地址",
        "status": "dealer",
        "conditions": ["C001"],
        "sig": "触发检测的交易签名"
    }
}
```

前端收到消息后，根据 sig 匹配交易记录，更新该钱包的用户标签。

## 九、常见问题

### 1. 为什么用 sortOrder=asc 获取最旧交易？
- SOL 链上最早的区块在最前面
- 庄家发币前通常需要准备操作，这些操作会留下痕迹

### 2. 为什么只检测第一笔交易？
- 第一笔交易最能反映钱包创建者的意图
- 后续交易可能被市场活动污染

### 3. 为什么需要 Redis 缓存？
- 避免重复检测同一个地址
- 减少 Helius API 调用
- 加快查询速度

### 4. 为什么用 asyncio.Semaphore？
- 控制并发检测数量
- 避免对 Helius API 造成过大压力
- 当前设置为 1（串行执行）
