# 庄家与散户判断逻辑说明

## 一、用户状态定义

用户状态存储在 Redis 的 `user:{address}` Hash 中，包含以下字段：

| 字段 | 说明 |
|------|------|
| `status` | 用户状态：`unknown`（未判定）、`dealer`（庄家）、`retail`（散户） |
| `conditions` | 满足的庄家条件列表，JSON 格式，如 `["C001", "C002"]` |
| `{mint}_holdingQty` | 当前持仓数量 |
| `{mint}_holdingCost` | 持仓成本 |
| `{mint}_avgPrice` | 平均价格 |

## 二、庄家检测条件

系统支持 5 个庄家检测条件，分为两类：

### 2.1 本地快速判断条件（C002-C005）

| 条件 | 说明 | 判断逻辑 |
|------|------|----------|
| **C002** | 使用 ALT（地址查找表） | `uses_lookup_table == True` |
| **C003** | Gas 费低于阈值 | `fee < dealer_gas_max`（默认 0.00001 SOL） |
| **C004** | CU（计算单元）在范围内 | `dealer_cu_min <= cu_consumed <= dealer_cu_max`（默认 0-200000） |
| **C005** | 风险分高于阈值 | `risk_score > dealer_risk_min`（默认 0） |

### 2.2 远程 API 判断条件（C001）

| 条件 | 说明 | 判断逻辑 |
|------|------|----------|
| **C001** | 首笔交易包含 closeAccount | 调用 Helius API 获取钱包首笔交易，检查是否包含 `closeAccount` 指令，且交易后该地址 SOL 余额为 0 |

### 2.3 条件启用控制

所有条件可通过后台设置开启/关闭：
- `dealer_c001_enabled` - C001 条件启用
- `dealer_alt_enabled` - C002 条件启用
- `dealer_gas_enabled` - C003 条件启用
- `dealer_cu_enabled` - C004 条件启用
- `dealer_risk_enabled` - C005 条件启用

## 三、判断流程

### 3.1 整体流程图

```
用户首次交易
    │
    ▼
状态初始化为 unknown
    │
    ├─── 本地快速判断（C002-C005）──→ 满足任一条件 → 判定为 dealer
    │
    ▼
入队等待 C001 检测
    │
    ├─── C001 未启用 → 保持 unknown（不标记为散户）
    │
    ├─── 调用 Helius API
    │       │
    │       ├── 成功 + 有 closeAccount → 判定为 dealer
    │       │
    │       └── 成功 + 无 closeAccount → 判定为 retail
    │
    └── 检测失败 → 重试（最多3次）→ 仍失败保持 unknown
```

### 3.2 详细判断步骤

#### 步骤 1：用户首次出现 → 初始化为 unknown

当用户（钱包地址）第一次出现在交易中时，创建 Redis 记录：
```python
state = {
    "status": "unknown",
    "conditions": []
}
```

#### 步骤 2：本地快速判断（C002-C005）

在 `trade_processor.py` 的 `get_trader_state()` 中同步执行：

```python
is_dealer, conditions = _check_local_dealer_conditions(tx_detail, state)

if is_dealer:
    state["status"] = "dealer"
    state["conditions"] = conditions
    await save_trader_state(redis, mint, address, state)
    # 立即排除出汇总
    await exclude_dealer(mint, address)
```

**判断结果：**
- 如果满足 C002/C003/C004/C005 中任一条件 → 立即标记为 **dealer**，无需等待远程 API
- 如果都不满足 → 继续下一步

#### 步骤 3：入队等待 C001 检测

如果本地判断不是庄家，将用户信息放入检测队列：

```python
await _enqueue_dealer_check(address, mint, sig)
```

队列消费者 `_dealer_queue_consumer()` 从队列取出任务，执行 `_run_dealer_check()`：

```python
# C001 未启用时，直接跳过
if c001_enabled != "true":
    return  # 保持 unknown 状态，不标记为散户

# C001 启用时，调用 Helius API
first_tx = await _fetch_first_transaction(address, api_key)

if first_tx and _check_c001(first_tx, address):
    conditions.append("C001")
    is_dealer = True

await _apply_dealer_result(redis, address, mint, sig, state, conditions, is_dealer)
```

#### 步骤 4：应用判定结果

```python
state["status"] = "dealer" if is_dealer else "retail"
await save_trader_state(redis, mint, address, state)

if is_dealer:
    await exclude_dealer(mint, address)  # 从汇总中排除
```

## 四、关键代码位置

| 功能 | 文件 | 函数/位置 |
|------|------|----------|
| 用户状态读写 | `trade_processor.py` | `get_trader_state()` |
| 本地条件判断 | `dealer_detector.py` | `_check_local_dealer_conditions()` |
| C001 远程检测 | `dealer_detector.py` | `_run_dealer_check()` |
| 结果应用 | `dealer_detector.py` | `_apply_dealer_result()` |

## 五、特殊情况处理

### 5.1 C001 未启用

当 `dealer_c001_enabled != "true"` 时：
- 直接跳过 C001 检测
- 用户状态保持 `unknown`
- **不会**被标记为散户

### 5.2 检测失败重试

检测失败时进入重试队列：
- 最多重试 3 次
- 等待时间指数退避：2s → 4s → 8s
- 3 次都失败则放弃，保持 `unknown` 状态
- 广播 `detection_failed` 给前端

### 5.3 重复检测防护

在队列消费者中，已标记为 `dealer` 或 `retail` 的用户会被跳过：
```python
if state.get("status") in ("retail", "dealer"):
    continue
```

## 六、庄家标记后的影响

当用户被判定为庄家时：

1. **状态更新**：Redis 中 `user:{address}` 的 `status` 变为 `"dealer"`
2. **汇总排除**：调用 `exclude_dealer()` 从 `metrics:{mint}` 汇总中减去该用户贡献
3. **WebSocket 广播**：通知前端用户状态变化
4. **交易跳过**：后续该用户的交易不会参与汇总计算

## 七、散户判定条件

**散户 = 未满足任何庄家条件的用户**

具体来说，以下情况会被判定为散户：
- C001 启用，且首笔交易**不包含** closeAccount
- 同时，本地条件 C002-C005 也都不满足
