## 二、核心模块说明

### 1. trade_stream.py - 实时交易流

**职责**：通过 Helius WebSocket 订阅实时交易

**主要流程**：
```
Helius WS ──► 接收消息 ──► 解析交易 ──► 入库 ──► 入队
                                                          │
                                                          ▼
                                                  _calculate_index()
                                                          │
                                    ┌─────────────────────┴─────────────────────┐
                                    ▼                                           ▼
                          get_trader_state()                          指数计算
                                    │                                           │
                                    ▼                                           │
                      如果 status=unknown → 自动入队检测                     │
                                                                          │
                                                                          ▼
                                                              ws_manager.broadcast("trade")
```

### 2. trade_backfill.py - 历史交易回填

**职责**：获取指定 mint 的历史交易

**主要流程**：
```
启动 ──► 等待 sync_point ──► 分页获取历史交易 ──► 批量入库
                                                    │
                                                    ▼
                                           _trigger_full_calculation()
                                                    │
                                    ┌───────────────┴───────────────┐
                                    ▼                               ▼
                          启动 dealer_consumer          run_full_calculation()
                                                                    │
                                                        ┌────────────┴────────────┐
                                                        ▼                         ▼
                                              _calculate_index()          指数计算
                                                        │
                                                        ▼
                                              get_trader_state() ──► 自动入队检测
```

### 3. trade_processor.py - 交易处理引擎

**职责**：指数计算和状态管理

**Redis 数据结构**：
```redis
# 用户状态 (user:{address})
user:ABC123...
  status: "unknown" | "dealer" | "retail"
  conditions: '["C001"]'
  {mint}_holdingQty: "1000"
  {mint}_holdingCost: "5.5"
  {mint}_avgPrice: "0.0055"
  {mint}_totalBuyAmount: "10"
  {mint}_totalSellAmount: "3"
  {mint}_totalSellPrincipal: "2.5"

# 指标汇总 (metrics:{mint})
metrics:XYZ789...
  total_bet: "50"
  realized_profit: "15"
  dealer_count: "3"
```

**核心方法 - get_trader_state：**
```python
async def get_trader_state(redis, mint: str, address: str, sig: str = None) -> dict:
    """
    获取用户状态
    如果 status=unknown 且有 sig，自动入队进行庄家检测
    """
```

**均价法计算逻辑**：
```
买入:
  holdingQty += buyQty
  holdingCost += buyAmount
  avgPrice = holdingCost / holdingQty

卖出:
  sellPrincipal = sellQty × avgPrice
  holdingQty -= sellQty
  holdingCost -= sellPrincipal
  realized = sellAmount - sellPrincipal
```

### 4. dealer_detector.py - 庄家判定服务

**职责**：识别并处理庄家账户

**判定条件 (C001)**：
- 首笔交易包含 `closeAccount` 指令
- 交易后该地址 SOL 余额为 0

**主要流程**：
```
队列消费 (_dealer_queue_consumer)
        │
        ▼
_run_dealer_check()
        │
        ├──► _fetch_first_transaction() ──► Helius RPC
        │
        ├──► _check_c001() ──► 判定条件
        │
        └──► 是庄家?
            │
            ├──► YES:
            │     save_trader_state(status="dealer")
            │     exclude_dealer()
            │     ws_manager.broadcast("user_status", {status: "dealer"})
            │
            └──► NO:
                  save_trader_state(status="retail")
                  ws_manager.broadcast("user_status", {status: "retail"})
```

## 三、数据流向

### 1. 启动流程

```
用户输入 mint ──► /admin/api/start
                        │
                        ▼
              TradeStream.start()  ←── 启动实时流 WS
                        │
                        ▼
              TradeBackfill.run()  ←── 启动历史回填
                        │
                        ▼
              等待 sync_point 就绪
                        │
                        ▼
              分页获取历史交易
                        │
                        ▼
              批量入库 + 触发计算
                        │
                        ▼
              dealer_consumer.start() ──► 启动庄家检测消费者
                        │
                        ▼
              consumer.start() ──► 启动 WS 消息消费者
```

### 2. 实时交易处理

```
Helius WS 推送交易
        │
        ▼
trade_stream._handle_message()
        │
        ├──► 入库 (Transaction)
        │
        └──► enqueue_trade() ──► 队列
                │
                ▼
        consumer._consumer_loop()
                │
                ▼
        _calculate_index()
                │
                ├──► get_trader_state() ──► 获取状态
                │                          │
                │                          └──► 如果 unknown → 自动入队检测
                │
                └──► 指数计算
                        │
                        └──► ws_manager.broadcast("trade") ──► 前端展示
```

### 3. 历史回填处理

```
TradeBackfill 批量入库
        │
        ▼
run_full_calculation()  ← 串行循环处理每条交易
        │
        └──► _calculate_index()
                │
                ├──► get_trader_state() ──► 自动入队检测
                │
                └──► 指数计算
                        │
                        └──► ws_manager.broadcast("trade")
```

### 4. 前端消息处理

```javascript
ws.onmessage = (evt) => {
  const msg = JSON.parse(evt.data);
  
  switch(msg.type) {
    case 'status':
      updateStatus(msg.data);  // 更新状态标签
      break;
      
    case 'trade':
      addTradeRow(msg.data);   // 添加交易行
      fetchMetrics();           // 刷新指标
      break;
      
    case 'user_status':
      handleUserStatus(msg.data);  // 处理用户状态更新
      // - dealer: 隐藏该用户所有交易行
      // - retail: 标记该用户所有交易行为"散户"
      break;
      
    case 'error':
      alert('错误: ' + msg.data.message);
      break;
  }
};
```

## 五、Redis 数据管理策略

### 用户数据 (user:{address})
- **生命周期**：永久保留，不随 mint 关闭而清除
- **目的**：记录用户的庄家/散户状态，跨 mint 复用

### 指标数据 (metrics:{mint})
- **生命周期**：随 mint 关闭而清除
- **目的**：仅用于当前 mint 的指数计算

### 为什么这样设计？
1. 庄家状态需要长期记录，同一地址在不同 mint 可能都是庄家
2. 不同 mint 的持仓数据独立，互不影响
3. 关闭 mint 时只需要清理指标，不需要清理用户状态

## 六、架构设计要点

### 庄家检测入口统一

**核心设计**：`get_trader_state()` 是唯一触发庄家检测的入口

```python
async def get_trader_state(redis, mint: str, address: str, sig: str = None) -> dict:
    # 如果是新用户且有 sig，自动入队检测
    if sig:
        await _enqueue_dealer_check(address, mint, sig)
    
    # 如果是 unknown 且有 sig，自动入队检测
    if status == "unknown" and sig:
        await _enqueue_dealer_check(address, mint, sig)
```

**好处**：
1. 单一职责：状态获取和检测触发在同一个方法内
2. 避免遗漏：所有调用 `get_trader_state` 的地方都会自动触发检测
3. 代码简洁：不需要在多处调用 `check_and_classify_user`

### 调用位置

| 位置 | 调用方式 |
|------|---------|
| `_calculate_index()` | 传入 sig，自动触发 |
| `_dealer_queue_consumer()` | 传入 sig，再次检查状态 |
