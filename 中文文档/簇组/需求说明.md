# 交易簇组（Cluster）完整模块设计方案
## 一、概述
我要把交易的  _extract_trade_info 中的信息  指令统计  一致的 作为 一个标准cu组 记录下来， 

基于交易特征  _extract_trade_info 中的指令统计信息   构建可配置、自动聚类、可人工锁定、可展示联动的簇组系统，用于批量识别、标记、管理散户/庄家交易，实现特征聚类 + 状态判定 + 人工终审的完整流程。

---

## 二、可配置匹配条件
支持单独启用/禁用 + 区间偏移量，所有开启条件必须同时满足才算匹配成功：
1. CU（计算单元）：基准值 ± 偏移量
2. 程序ID数量：基准数 ± 偏移量
3. 主指令数量：基准数 ± 偏移量
4. 内部指令数量：基准数 ± 偏移量

---

## 三、簇组结构 redis key 簇组名称
1. 标识信息
   - 簇组名称（首个钱包地址）   
   - 所属文件夹 默认为空

2. 状态信息
   - 是否启用
   - 簇组类型：未定义 / 散户 / 庄家
   - 判定类型：系统判定 / 手动锁定

3. 基准特征
   - 基准CU、程序ID数、主指令数、内部指令数， 

4. 成员数据 去重
   - 所属Tx列表
   - 所属用户（钱包地址）列表

5. 统计数据
   - 总Tx数
   - 总独立用户数

---

## 四、系统自动判定庄家规则（可配置阈值）
- 在簇组设置中自定义：Tx数阈值、用户数阈值（默认均＞50）
- 满足双阈值 → 系统自动标记为庄家簇组
- 仅对未经过手动修改的簇组生效

---

## 五、Tx 与用户关联逻辑
1. 每笔Tx匹配簇组后，记录自身所属簇组信息
2. 每个用户（钱包）自动绑定其所属簇组
3. 实时交易列表展示：
   - 簇组缩写、ID、名称
   - 簇组状态（未定义/散户/庄家）
4. 点击交易列表中的簇组信息，可直接修改簇组状态

---

## 六、手动锁定规则（核心）
1. 簇组类型支持两种来源：
   - 系统判定：按阈值自动判断
   - 手动调整：人工设为散户/庄家
2. 一旦手动修改过簇组类型，系统后续不再自动覆盖，保持锁定
3. 未手动修改的簇组，系统可按规则自动更新类型

---

## 七、运行时处理流程
1. 新Tx进入，遍历所有已启用簇组进行特征匹配
2. 匹配成功 → 加入该簇组，Tx与用户绑定此簇组，继承簇组状态
3. 无匹配 → 新建簇组，以首个钱包地址命名
4. 簇组数据达到阈值 → 系统自动标记庄家（未手动锁定时生效）
5. Tx和用户均带上所属簇组信息，在列表中可查看与操作

---

## 八、簇组管理页面功能
1. 查看所有簇组：基本信息、状态、启用开关、Tx数、用户数、判定类型
2. 查看簇组内所有Tx详情（同实时交易页面样式）
3. 支持修改：
   - 启用状态
   - 所属文件夹
   - 簇组类型（手动锁定）
4. 手动修改后锁定，系统不再自动变更
